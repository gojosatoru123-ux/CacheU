What are the three prerequisites before installing the project?

<!-- ANSWER -->
1. **Node.js v18 or higher** ŌĆö check with `node --version`
2. **pnpm v8 or higher** ŌĆö check with `pnpm --version`
3. **A modern browser** ŌĆö Chrome, Firefox, Safari, or Edge

What is the exact sequence of commands to clone the project and start the dev server?

<!-- ANSWER -->
```bash
# 1. Clone the repository
git clone https://github.com/your-org/my-project.git
cd my-project

# 2. Install all dependencies
pnpm install

# 3. Copy environment variables
cp .env.example .env

# 4. Start the development server
pnpm dev
```

The dev server starts on `http://localhost:3000` by default.

<!-- QUESTION -->
difficulty: Easy
tags: environment, config

What file contains environment variables, and why should `.env` never be committed to version control?

<!-- ANSWER -->
Environment variables live in the `.env` file (copied from `.env.example`).

**Why `.env` must never be committed:**
- It contains secrets: API keys, database passwords, JWT secrets
- If pushed to GitHub (even in a private repo), keys can be scraped by bots
- Different environments (dev/staging/prod) need different values

The `.gitignore` file should always include:
```
.env
.env.local
.env.*.local
```

The `.env.example` file IS committed ŌĆö it shows which variables are needed but with dummy/empty values.
<!-- END -->

<!-- QUESTION -->
difficulty: Medium
tags: project-structure, architecture

Describe the purpose of each top-level folder: `src/`, `public/`, and `dist/`.

<!-- ANSWER -->
| Folder | Purpose |
|---|---|
| `src/` | All source code ŌĆö components, pages, utilities, styles |
| `public/` | Static assets served as-is (favicons, robots.txt, images that don't need bundling) |
| `dist/` | **Build output** ŌĆö generated by `pnpm build`, never edit manually, gitignored |

**Key distinction:** Files in `public/` are copied verbatim. Files in `src/` go through the bundler (Vite/Webpack) ŌĆö TypeScript compilation, import resolution, code splitting, minification.
<!-- END -->

<!-- QUESTION -->
difficulty: Medium
tags: build, deployment

What does `pnpm build` do differently from `pnpm dev`? When would you run each?

<!-- ANSWER -->
| Command | Use case | Output |
|---|---|---|
| `pnpm dev` | Development ŌĆö hot reload, source maps, no minification | Dev server on localhost |
| `pnpm build` | Production ŌĆö optimized, minified, tree-shaken bundle | Static files in `dist/` |

**`pnpm build` specifically:**
- Compiles TypeScript and checks for type errors
- Bundles all imports into optimized chunks
- Minifies JS and CSS (removes whitespace, shortens variable names)
- Generates content-hashed filenames for cache busting
- Tree-shakes unused code

Run `pnpm build` before deploying to any production or staging environment.
